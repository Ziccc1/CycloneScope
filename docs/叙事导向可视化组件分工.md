# CycloneScope 叙事导向可视化组件分工

## 1. 文档目的

本文件规定 CycloneScope 剩余叙事型页面与可视化组件的负责人、组件边界、输入输出和组织方式。

本轮开发采用“一个组件、一名负责人、完整交付”的原则。组件负责人独立完成组件所需的数据读取、数据转换、界面呈现、局部状态、交互行为、空状态、错误状态和验证，不把一个组件拆成多人共同实现。其他成员只负责提供已经约定的上游数据或公共接口，不进入该组件内部继续开发。

跨组件联动统一通过共享状态和稳定的数据类型完成。地图组件不直接读取其他图表内部状态，图表也不直接调用地图实例。组件只读取共享状态，并通过明确的回调更新共享状态。

## 2. 最终叙事结构

CycloneScope 的叙事落点不是某一个地区的设施规划，而是解释热带气旋如何从全球尺度的自然现象，转化为具体事件、灾害影响和区域响应问题。台湾是区域影响分析的数据完整案例，不是整个系统唯一关心的对象。

项目围绕以下核心问题展开：

> 全球热带气旋具有怎样的时空分布和演化差异？一场气旋如何形成具体影响？当分析尺度深入到数据较完整的地区时，我们又能如何识别风险并比较响应方案？

整体叙事由六个连续层次构成：

1. 全球历史概览：认识气旋在时间、海盆、季节和强度上的整体分布；
2. 单场事件分析：从全球档案中进入一场具体气旋，观察轨迹、强度和风场的动态过程；
3. 历史相似性：通过路线匹配把单场事件放回历史样本中，理解“路径相似”和“影响相同”之间的区别；
4. 从危险到灾害：比较强度、暴露和报告灾损，说明自然危险必须与人口和区域条件结合才能形成灾害影响；
5. 台湾区域案例：由于当前数据在台湾具有更细的人口、行政区和设施粒度，进一步展示区域风险如何被识别；
6. 响应方案比较：在区域案例中检验设施覆盖，并比较调整方案带来的变化。

这条叙事保持从全球到区域、从现象到影响、从理解到行动的尺度递进。区域案例用于证明全球分析框架可以继续深入，而不是反过来把全球模块降格为台湾分析的背景。

### 2.1 第一层：全球历史概览

故事从全球热带气旋档案开始。全球地图展示1980年以来的历史轨迹，并配合年度数量、月份分布、海盆构成和强度分布，使观众首先建立整体认识：热带气旋并非均匀分布，不同海盆具有不同的活动范围、季节节律和强度结构。

这一层需要回答的是“气旋主要出现在哪里、什么时候更活跃、不同海盆有何差异”。地图负责空间格局，统计组件负责把大量轨迹归纳为可比较的时间和类别结构。筛选海盆、年份和强度时，地图与统计图同步变化。

这一层不能在没有统计检验的情况下宣称气旋正在变多、变强或由某个气候因素导致。IBTrACS早期观测完整性、不同机构记录差异和时间范围必须在方法说明中保留。全球概览的结论应限定为当前数据范围内的分布特征。

全球概览的转场来自“整体分布由一场场具体事件构成”。用户可以从地图、年份峰值、海盆或经典案例库选择任意一场事件，进入单场分析，而不是被预设为只能选择莫拉克。

### 2.2 第二层：单场事件分析

单场分析解释一场热带气旋如何沿轨迹移动并发生强度变化。页面以轨迹地图、既有ERA5风场、时间播放和台风强度时间曲线为核心，显示当前位置、风速、气压、ACE、持续时间和观测来源。

该层关注过程，而不是立即讨论台湾设施。用户可以选择不同海盆和年份的事件，观察轨迹方向、移动速度、强度变化和风场结构。地图与强度曲线通过统一时间状态同步，使单个最大风速不再成为理解事件的唯一指标。

单场分析应让用户形成两个认识：第一，气旋是动态过程，位置、持续时间和强度共同变化；第二，即使最大强度接近，不同轨迹和移动过程也可能带来不同影响范围。

这一层自然引出历史比较问题：当前事件在历史档案中是否存在相似路径，它们除了路径之外还有哪些差异？

### 2.3 第三层：历史相似性与路线匹配

路线匹配是连接全球档案与单场事件的重要分析方法，而不是附属的绘图功能。用户可以使用当前事件的完整或部分轨迹，也可以手绘一条关注路径，从全球历史记录中检索地理位置、形状和行进方向相似的事件。

地理相似用于寻找经过相近区域的历史事件，形状相似用于寻找跨海盆但具有相似转向结构的事件。匹配结果同时展示路径相似度以及最大风速、最低气压、ACE、持续时间和移动速度等差异。

这一层的核心认识是：路线匹配用于寻找值得比较的历史案例，而不是预测未来结果。路径相似并不意味着强度、风场、人口暴露或灾损相同。相似案例之间的差异将叙事推进到下一个问题——自然危险如何转化为灾害影响。

### 2.4 第四层：从自然危险到灾害影响

该层将轨迹和强度分析扩展到影响比较。页面区分三种语义：气象危险表示风速、降水或危险度等物理条件；人口暴露表示处在危险范围内的人口估计；报告灾损表示数据源记录的死亡、受灾人数或经济损失。

经典事件比较组件用于展示最大风速、持续时间、模型暴露和报告灾损之间并非简单对应。系统不把报告灾损与模型暴露合成一个总分，也不根据风速直接推断实际损失。缺少影响数据的事件保留为空值并说明数据能力。

这一层的核心认识是：气旋强度描述自然现象，灾害影响还取决于路径经过的位置、人口暴露、区域脆弱性和数据记录范围。全球数据可以支持事件级和国家级比较，但要进一步观察行政区内部的差异，需要进入具有更细数据粒度的区域案例。

### 2.5 第五层：台湾区域案例

台湾在叙事中承担“区域深描”的角色。当前项目在台湾具备行政区、人口、危险度和避难、医疗、救援设施等较完整数据，因此可以把全球层面的分析方法继续推进到行政区和设施尺度。

用户从具备台湾影响数据的历史事件中选择一个案例，例如莫拉克、海燕或其他可用事件。页面必须显示历史事件年份、人口参考年份和设施数据时间，明确当前结果属于历史危险情景与当前区域数据的组合分析，不等同于历史实际受灾复原。

地图、风险指标摘要和行政区风险排名共同回答：危险程度较高的区域在哪里、暴露人口主要集中在哪里、危险度最高与暴露人口最多的区域是否一致。结论由当前事件、指标和阈值实时生成，不预先固定为某个行政区。

台湾案例证明的是同一套分析框架如何从全球事件深入到区域内部，而不是将全球数据强行解释为台湾问题。

### 2.6 第六层：区域响应与方案比较

在识别区域风险之后，叙事进入响应层面。设施图层、覆盖构成和缺口诊断表用于检查高风险人口是否处于避难、医疗或救援设施的服务范围和容量之内。

用户可以新增、移动或删除模拟设施，系统重新计算覆盖范围、容量分配和未覆盖人口。建设前后比较保持相同地图范围、风险阈值和色标，显示覆盖率、未覆盖人口、高风险盲区和预算点的变化。

不同设施类型分别分析，避难容量、医疗能力和救援单位不得直接相加。系统也不把某个模拟方案称为真实最优选址，而是展示不同目标下的相对改善与剩余缺口。

这一层将整个故事从观察和解释推进到方案比较，但最终仍回到全球分析框架：CycloneScope的价值在于把全球历史、单场过程、相似案例、灾害影响和区域响应连接在同一套多尺度可视分析系统中。

### 2.7 页面中的叙事组织方式

叙事不采用固定答辩脚本，也不要求专门的演示预设。用户可以按照“全球概览—单场事件—相似案例—影响比较—台湾案例—响应方案”的推荐顺序探索，也可以从任意模块进入自由分析。

每一层页面只需要稳定表达三类信息：当前尺度正在回答的问题、当前数据支持的发现、进入下一尺度的原因。若需要自然语言摘要，由后端LLM接口根据已经计算并验证的结构化指标生成，前端只显示返回结果，不再单独开发故事容器或叙事卡组件。

LLM不得直接读取原始大数据自行计算，也不得补全缺失数字。请求中应提供当前筛选、指标名称、单位、数据状态、允许引用的数值和限制说明；返回内容只负责组织语言。接口不可用时页面仍应依靠图表标题、指标标签和方法说明完成分析，不影响核心可视化使用。

页面顶部可保留轻量的尺度导航：

```text
全球格局 → 单场过程 → 历史比较 → 灾害影响 → 区域案例 → 响应方案
```

该导航用于说明分析尺度和当前位置，不将页面锁定为只能按固定顺序播放。全球概览始终是系统的起点和独立分析模块，台湾案例则是全球框架向区域粒度深入后的一个数据完整实例。

## 3. 公共协作规则

组件负责人对组件交付结果负完整责任，包括：

- 组件源文件与样式；
- 数据读取和组件内部派生计算；
- 组件内的筛选、选择、悬停、展开和切换交互；
- 加载、空数据、错误和不可用状态；
- 单位、图例、文本、数值精度和无障碍标签；
- 与共享状态之间的输入输出；
- 必要的单元测试或交互验证；
- 在答辩分辨率下的视觉检查。

如果组件需要新增公共字段，负责人先在接口说明中提出完整字段需求，由 B 统一维护公共 Schema 和生成类型。Schema 生成完成后，组件负责人继续完成自己的组件，不将组件剩余工作转交给 Schema 维护者。

公共状态只保存跨组件共享的选择，不保存图表内部的临时视觉状态。建议公共状态包含：

```ts
type StoryStage =
  | "global"
  | "event"
  | "similarity"
  | "impact"
  | "regional"
  | "response";

type SharedAnalysisState = {
  storyStage: StoryStage;
  selectedBasin: string | null;
  selectedYearRange: [number, number];
  selectedStormId: string | null;
  selectedZoneId: string | null;
  currentTime: string | null;
  selectedImpactMetric: string;
  selectedFacilityType: string;
  selectedScenarioId: string | null;
  scenarioView: "baseline" | "current";
  selectedMatchId: string | null;
};
```

任何组件都不得建立第二套与上述字段含义相同的全局状态。

## 4. A 负责的组件

### 4.1 台风强度时间曲线

组件建议命名为 `StormIntensityChart.tsx`。

该组件用于重演单场台风的发展过程，完整展示时间、风速与气压之间的变化关系。组件由 A 独立负责，包括读取单场轨迹时间序列、整理风速和气压、处理缺失值、绘制共享时间轴的上下两组曲线、显示当前时间指示线、标注最大风速与最低气压、标注最接近台湾的时刻，并完成 Tooltip、图例和空状态。

组件读取 `selectedStormId` 和 `currentTime`，在用户点击或拖动曲线时更新 `currentTime`。组件只更新共享时间，不直接控制地图、粒子或底部时间轴。其他组件通过同一个 `currentTime` 自动响应。

曲线中的未知值保持断线，不以零代替。风速统一使用 `m/s`，气压统一使用 `hPa`。切换事件时，组件必须清理上一事件的派生状态并重新定位有效时间。

### 4.2 行政区风险排名

组件建议命名为 `ZoneRiskRanking.tsx`。

该组件用于回答哪些行政区最值得优先关注。A 独立负责行政区危险度、暴露人口、未覆盖人口和覆盖率的整理、排序、展示与解释。组件提供“危险度最高”“暴露人口最多”“未覆盖人口最多”三个排名视角，并显示行政区名称、原始数值、单位和当前指标口径。

用户选择排名条目时，组件更新 `selectedZoneId`。组件不直接调用 MapLibre，也不负责地图飞行或高亮。地图通过共享的 `selectedZoneId` 响应。

该组件同时负责指标定义说明，明确区分危险度、人口暴露、设施覆盖和报告灾损。数据不足的行政区不得进入需要该字段的排名，页面应说明排除原因和数量。

### 4.3 路线候选事件比较图

组件建议命名为 `MatchedStormComparison.tsx`。

该组件用于比较路线匹配候选在路径之外的差异。A 独立负责读取匹配结果中的事件标识，连接最大风速、最低气压、ACE、持续时间、平均移动速度、发生月份和台湾分析能力等字段，并以标准化条形图或紧凑比较表完成呈现。

组件必须保留原始数值，不只展示标准化结果。组件需要持续显示“路径相似不代表强度和影响相同”的解释。用户选择候选事件时，组件更新 `selectedMatchId`，但不负责在地图中绘制候选轨迹，也不负责载入风险情景。

### 4.4 数据语境与分析限制

组件建议命名为 `ScenarioDataContext.tsx`。

该组件用于说明历史危险情景与当前人口、设施数据之间的时间关系。A 独立负责读取数据版本、参考年份、数据状态和来源信息，并展示历史台风年份、人口参考年份、设施数据时间、轨迹来源、风场来源和情景性质。

组件必须明确说明当前分析属于“历史危险情景作用于当前人口与设施的反事实压力测试”，不是对历史真实受灾情况的复原，也不是实时预报或真实工程选址结论。该组件以紧凑摘要和可展开说明两种状态呈现，并由 A 完成全部展开交互和内容维护。

### 4.5 风险指标摘要

组件建议命名为 `RiskMetricCards.tsx`。

该组件用于概括当前历史情景作用于当前人口后产生的风险结果。A 独立负责指标统计、数据读取、数值格式、卡片布局和选择交互，完整展示高危险区域数量、高风险人口、已覆盖高风险人口、未覆盖高风险人口、覆盖率、风险最高行政区、缺口最大行政区和当前危险阈值。

组件必须保证每个数字都能追溯到当前事件、当前影响指标、当前阈值和当前设施类型。用户点击行政区指标时，组件更新 `selectedZoneId`。组件自行处理加载、错误、空数据和字段缺失状态，不直接操作地图或地图图层。

### 4.6 设施覆盖构成

组件建议命名为 `CoverageBreakdown.tsx`。

该组件用于展示高风险人口中的已覆盖人口、未覆盖人口、服务范围外人口和容量不足人口。A 独立负责读取当前评估结果、核对指标口径、计算展示比例、绘制水平构成条、显示数量与百分比，并完成设施类型和基线/当前方案的组件内切换。

不同设施类型必须分别展示，不得把避难、医疗和救援容量直接相加。容量未知时，组件必须显示数据限制，不能推断为零。组件读取 `selectedFacilityType` 和 `scenarioView`，但不直接切换地图图层。

### 4.7 建设前后指标比较

组件建议命名为 `ScenarioMetricComparison.tsx`。

该组件用于比较基线与当前设施方案。A 独立负责保存并读取两组评估摘要，核对各项指标口径，展示覆盖率、未覆盖人口、高风险盲区数量、容量利用率和预算点的基线值、当前值及变化值，并完成“建设前”“建设后”和“恢复基线”的界面交互。

组件通过更新 `scenarioView` 控制全局当前查看状态，但不直接修改地图图层。地图根据 `scenarioView` 展示对应结果。恢复基线操作、加载反馈、成功反馈、失败回滚和空方案状态均由A在该组件中完整实现。

## 5. B 负责的组件

### 5.1 全球气旋统计概览

组件建议命名为 `GlobalCycloneOverview.tsx`。

该组件与全球历史轨迹地图共同构成系统起点。B 独立负责当前年份、海盆和强度筛选下的汇总计算、四张协调统计图、筛选交互、布局、空状态和验证。年度、月份、海盆和强度视图必须共享同一筛选范围，并与地图状态同步。

### 5.2 全球海盆对比

组件建议命名为 `BasinComparison.tsx`。

该组件用于比较不同海盆的季节结构、事件数量、最大风速分布、ACE分布和典型持续时间。B 独立负责海盆选择、指标标准化、比较图表、Tooltip、数据范围说明和筛选交互。用户可以同时选择两至四个海盆，组件不把样本数量差异误写成发生概率差异。

### 5.3 全球事件影响比较

组件建议命名为 `StormImpactComparison.tsx`。

该组件用于连接自然危险与灾害影响。B 独立负责将经典事件的最大风速、持续时间、ACE、模型暴露、报告死亡、报告受灾人数和名义经济损失按来源和语义分组呈现，并完成事件选择、指标切换、缺失值表达和数据来源说明。

组件不得把模型暴露和报告灾损合并为同一总分，也不得将缺失值显示为零。名义经济损失必须保留原始货币年份说明。用户选择事件时更新 `selectedStormId`，从而可以进入该事件的单场过程或区域案例。

### 5.4 多台风综合对比

组件建议命名为 `StormComparisonCharts.tsx`。

该组件用于让用户从全球地图、经典案例库或路线匹配结果中选择两至三场台风，并通过一组联动图表进行并列比较。B 独立负责比较集合管理、轨迹小倍图、生命周期强度曲线、核心指标点图、灾害影响条形图、事件移除与替换、统一数值格式、缺失字段说明和选择状态。

比较图表共同使用固定的事件颜色，并通过悬停和选择联动。标量指标包括最大风速、最低气压、ACE、持续时间、移动速度和路径长度；模型暴露与报告灾损放在独立影响图中，不与气象指标合成总分。

### 5.5 设施缺口诊断表

组件建议命名为 `CoverageGapTable.tsx`。

该组件用于呈现行政区层面的设施覆盖缺口。B 独立负责读取评估结果、组织表格字段、完成排序、设施类型切换、行选择、缺失容量说明和表格滚动。

表格展示行政区、高风险人口、已覆盖人口、未覆盖人口、覆盖率和主要缺口。用户选择表格行时更新 `selectedZoneId`。表格需要明确当前覆盖方法、人口参考年份、设施类型和容量数据状态。

### 5.6 路线匹配结果列表

组件建议命名为 `TrajectoryMatchList.tsx`。

该组件用于展示 Top 3 或 Top 5 历史匹配案例。B 独立负责候选列表、综合相似度、地理分量、形状分量、方向分量、数据能力标签以及候选选择交互。

每个候选提供“查看轨迹”和“载入风险情景”。查看轨迹时更新 `selectedMatchId`。载入风险情景时，组件完成数据能力检查、事件状态切换、故事阶段切换、加载状态和错误提示。缺少台湾危险数据的候选不允许进入设施分析，并在按钮旁说明原因。

## 8. 组件视觉与交互设计说明

### 8.1 全球气旋统计概览

`GlobalCycloneOverview.tsx` 采用一个二乘二的小倍图布局，与全球地图同时出现在概览阶段。左上为年度气旋数量折线图，横轴是年份，纵轴是事件数量；折线下方使用低透明度面积填充，用于帮助观察相对高低，但不额外绘制未经验证的趋势拟合线。用户在年度图上拖动刷选区间，更新 `selectedYearRange`，地图和其余三个统计图同步重新汇总。

右上为十二个月份柱状图，固定从一月至十二月排列，柱高表示当前筛选范围内的事件数量。月份图使用统一的蓝绿色，不把月份误编码为强度等级。悬停显示数量、占比和当前海盆；点击月份可以作为组件内部的临时高亮，但若月份需要跨组件筛选，应先把月份字段纳入公共状态。

左下为海盆水平条形图，按照事件数量从高到低排列。每个海盆保持固定颜色映射，并与地图筛选控件一致。点击条形更新 `selectedBasin`；再次点击当前海盆恢复全球。右下为最大风速分布直方图，使用固定风速分箱，横轴为 `m/s`，纵轴为事件数量，用于观察强度结构，不使用饼图或玫瑰图代替定量比较。

四张图共享标题区，标题区显示事件总数、时间范围和当前海盆。组件默认高度控制在右侧或底部分析面板可见范围内；屏幕较窄时改为两列滚动，不压缩坐标标签。无数据时保留坐标框架并显示“当前筛选无事件”，不显示全零柱形制造数据存在的错觉。

### 8.2 台风强度时间曲线

`StormIntensityChart.tsx` 使用上下对齐的双折线小倍图，而不是把风速和气压强行放在一个双轴图中。上图显示风速，使用青绿色实线；下图显示气压，使用暖灰色或橙色实线。两张图共享时间轴、当前时间垂直指示线和同一个悬停位置，因此用户可以比较风速上升与气压下降是否同时发生。

风速曲线可在背景中使用非常浅的强度等级分区，但等级不适用或类别未知时不显示分区。最大风速点使用实心标记，最低气压点使用空心标记，最接近台湾或登陆时刻使用带文字的垂直参考线。缺失值通过断线表现，断线悬停时显示“该时刻无观测”，不能进行线性补值后当作真实记录呈现。

鼠标移动时显示统一 Tooltip，内容包含UTC时间、风速、气压、气旋位置和数据来源。单击曲线或拖动时间指示线更新 `currentTime`。为了避免与底部播放条竞争，曲线只负责精确选择和观察变化，播放、暂停和速度仍由既有时间控制负责。

### 8.3 行政区风险排名

`ZoneRiskRanking.tsx` 使用水平条形排名图。顶部以分段按钮切换“危险度”“暴露人口”“未覆盖人口”和“覆盖率”，每次只展示一种明确指标，避免把单位不同的数据堆在同一坐标轴。行政区名称放在纵轴，条形从统一零基线开始，右端直接标注格式化数值。

默认显示前十个行政区，并在组件底部提供“显示全部”展开。当前地图选中的行政区即使不在前十，也以一条固定在底部的对照行显示。点击条形更新 `selectedZoneId`；选中条形增加描边和左侧定位标记，其他条形降低透明度但仍保留比较背景。

危险度使用与地图危险图层一致的连续色带；人口类排名使用单一颜色深浅，不与危险等级混淆；覆盖率使用中性色到蓝绿色的顺序色带。排序切换时使用短动画保持对象连续性，但动画不得改变数值尺度。数据缺失的行政区在图下方以文字说明，不绘制长度为零的条形。

### 8.4 路线候选事件比较图

`MatchedStormComparison.tsx` 使用标准化点图与原始值表格的组合。上半部分为多指标点图，每一行代表最大风速、最低气压、ACE、持续时间和移动速度，每一列范围标准化到零至一，不同候选事件用固定颜色的圆点和连线表示。点图用于观察相对差异，不把标准化数值解释为新的综合评分。

下半部分为紧凑原始值表，列为候选事件，行与点图指标一致，显示 `m/s`、`hPa`、小时或天数等真实单位。发生月份、海盆和台湾数据能力以文本与标签呈现，不参与数值标准化。用户选择某一候选时更新 `selectedMatchId`，对应点、表格列和地图候选轨迹使用相同高亮色。

组件标题区持续显示当前匹配模式和“路径相似不代表影响相同”。如果候选少于三场，图表按实际数量绘制；字段缺失时点图中留空，表格显示破折号并在 Tooltip 中说明缺失来源。

### 8.5 全球事件影响比较

`StormImpactComparison.tsx` 以散点图作为主图，回答气旋强度与灾害影响是否简单对应。横轴默认使用最大风速，也可以切换ACE或持续时间；纵轴可以在模型暴露人口、报告死亡、报告受灾人数和名义经济损失之间切换。每个点代表一场经典事件，颜色编码海盆，点形状区分模型数据和报告数据的可用状态。

人口、死亡和经济损失跨事件差异较大时使用对数轴，并在轴标题中明确标注；零值与未知值严格区分，未知值不进入散点位置。悬停显示事件名称、年份、海盆、横纵轴原始值、数据状态和来源。点击点更新 `selectedStormId`，使地图定位到对应事件。

散点图下方增加一个按当前纵轴指标排序的迷你水平条形图，用于在点重叠或样本较少时提供精确比较。组件不绘制自动回归线，除非后续确实进行了统计检验并在页面说明方法。报告灾损和模型暴露永远通过指标切换分别显示，不在同一纵轴相加。

### 8.6 数据语境与分析限制

`ScenarioDataContext.tsx` 采用“时间语境条＋展开说明”的形式。紧凑状态下横向显示历史事件年份、风场时间、人口参考年份、设施参考时间和分析性质，中间使用箭头表达数据组合关系。例如“2009历史危险情景 → 2024/2025人口与当前设施”，让时间错位在进入区域分析时立即可见。

展开后使用三列信息块分别说明自然危险、人口暴露和设施响应的数据来源、状态与限制。观测、再分析、模型估计和报告数据使用文字标签而不是只靠颜色区分。组件不使用复杂图表，因为它的任务是保证解释口径清楚；视觉重点是时间、来源和语义的对应关系。

### 8.7 风险指标摘要

`RiskMetricCards.tsx` 使用两行指标卡布局。第一行显示高风险人口、已覆盖人口、未覆盖人口和总体覆盖率；第二行显示危险度最高行政区、暴露人口最高行政区、缺口最大行政区和当前阈值。人口数字使用统一缩写规则，卡片内同时保留完整数字的Tooltip。

覆盖类卡片使用蓝绿色，缺口类卡片使用橙红色，危险度卡片使用地图当前危险色带中的代表色。颜色只作为语义辅助，卡片始终显示明确标签和单位。点击包含行政区的卡片更新 `selectedZoneId`；选中后卡片增加描边，并与地图和排名图联动。

卡片不使用仪表盘。覆盖率使用数字和一条细进度线表达，避免占用大量空间。数据加载时使用保持卡片尺寸的骨架屏；数据未知时显示“暂无数据”，不能继承上一个事件的数值。

### 8.8 设施覆盖构成

`CoverageBreakdown.tsx` 使用百分比堆叠条与绝对人数说明。主条形总长度代表当前设施类型下的全部高风险人口，依次分为已覆盖、服务范围外和容量不足三个互斥部分。条形下方分别标注人数和百分比，保证颜色面积与文字数值同时可读。

基线与当前方案比较时使用上下两条完全相同尺度的堆叠条，上方为基线，下方为当前方案；新增覆盖的部分可以使用短暂高亮描边，但不改变类别颜色。设施类型通过组件顶部的分段按钮切换，避难、医疗和救援分别重新计算并重新绘制。

当容量字段未知、覆盖算法只支持距离或某类设施没有有效数据时，组件在条形上方显示方法标签，例如“服务半径估计”或“容量未纳入”。无法拆分服务范围外与容量不足时，只显示已覆盖和未覆盖两部分，不能编造细分类别。

### 8.9 设施缺口诊断表

`CoverageGapTable.tsx` 使用可排序表格，并在数值列中嵌入数据条。主要列为行政区、高风险人口、已覆盖人口、未覆盖人口、覆盖率和主要缺口。未覆盖人口列中的浅红数据条帮助快速比较，但真实数字仍直接显示在单元格中。

表头点击控制排序，默认按照未覆盖人口从高到低。设施类型切换位于表格顶部，切换后整张表使用该类型的独立评估结果。点击表格行更新 `selectedZoneId`，选中行使用与地图行政区高亮一致的描边色。

表格不试图把所有字段塞进一屏。覆盖方法、容量质量和人口参考年份放在表格标题下方的语境行；详细来源放入行展开区。容量未知的记录显示“容量未知”，不能按零容量排序到最差位置。

### 8.10 建设前后指标比较

`ScenarioMetricComparison.tsx` 使用哑铃图比较基线与当前方案。每一行代表覆盖率、未覆盖人口、高风险盲区、容量利用率和预算点，左点为基线，右点为当前方案，两点之间用线连接，并在右侧直接标注变化值。

由于指标单位不同，每一行使用自己的明确刻度或采用数值卡加变化箭头，不能把所有指标未经说明标准化后放进同一公共坐标轴。有利变化使用蓝绿色，不利变化使用橙红色；对于“未覆盖人口”和“盲区数量”这类越低越好的指标，颜色判断必须反向处理。

组件顶部提供“建设前”“建设后”地图视图切换，更新 `scenarioView`。恢复基线作为次要按钮放在组件底部，并在执行后显示保存或失败反馈。没有模拟设施时，当前方案与基线重合，组件显示“尚未产生方案变化”，而不是绘制虚假的零增益。

### 8.11 路线匹配结果列表

`TrajectoryMatchList.tsx` 使用纵向候选卡列表，默认显示Top 5。每张卡片左侧为排名与事件名称，中间为综合相似度横条，右侧为年份、海盆和数据能力标签。卡片展开后显示地理、形状和方向三个分量的迷你条形，并展示可进入的后续分析模块。

综合分数使用零至一百的统一尺度，但明确标注为相似度而非概率。地理、形状和方向分量采用相同长度尺度，避免仅通过颜色深浅比较。点击卡片更新 `selectedMatchId` 并让地图高亮轨迹；“载入事件”将该事件设置为 `selectedStormId`。只有具备台湾区域数据的事件显示“进入台湾案例”，其他事件仍可进入单场和影响比较。

数据能力标签使用文字缩写与图标组合，例如“风场”“影响”“台湾区域”，不能只用绿色圆点表示。请求中显示局部加载状态，失败时保留用户手绘路径和上一次有效输入，方便重试。

### 8.12 全球海盆对比

`BasinComparison.tsx` 采用“季节热力图＋指标点图”的上下布局。上半部分为海盆×月份热力图，行表示用户选择的海盆，列固定为十二个月，颜色表示该海盆在当前年份范围内各月事件占本海盆全年事件的比例。使用比例而不是原始数量，可以避免样本量较大的海盆在所有月份都显得更强；Tooltip同时显示比例和原始事件数。

下半部分为海盆指标点图，每一行分别表示年均事件数、最大风速中位数、ACE中位数和持续时间中位数，每个海盆用固定颜色圆点表示。各指标使用自己的原始坐标与单位，不绘制没有可比意义的总排名。海盆选择器允许同时比较两至四个海盆，颜色与全球地图和概览条形图保持一致。

组件可以增加强度分布箱线图作为展开视图，比较中位数、四分位数和异常值。箱线图必须同时显示样本数，避免用少量事件与大量事件进行无语境比较。用户点击海盆名称更新 `selectedBasin`，但多海盆比较集合保留为组件内部状态，不覆盖地图当前单海盆筛选，除非后续扩展公共状态。

### 8.13 多台风综合对比

`StormComparisonCharts.tsx` 由四个共享事件颜色和选择状态的图表组成，不使用表格代替主要比较，也不使用容易受面积和角度影响的雷达图。

第一张图是轨迹小倍图。每场台风使用相同尺寸的地图框，保留真实地理位置，并显示起点、终点、行进方向和按风速编码的轨迹线。小倍图默认使用各自适合观察的视口，同时在标题中显示比例尺；用户可以切换“独立视口”和“统一视口”。统一视口适合同海盆地理位置比较，独立视口适合跨海盆形态比较。

第二张图是生命周期强度曲线。由于不同台风持续时间不同，横轴默认归一化为生命周期百分比零至一百，纵轴为风速 `m/s`，两至三条事件曲线叠加比较，并标出各自最大风速点。用户可以切换到真实UTC时间视图；只有时间范围存在交集时才适合直接按UTC叠加。气压以第二个上下对齐的小图展示，不与风速共用双轴。

第三张图是核心指标对齐点图。每一行对应最大风速、最低气压、ACE、持续时间、移动速度和路径长度，每个指标使用自己的水平标尺，不同台风以固定颜色圆点表示，并在点旁显示原始值。它用于准确比较单值指标，而不是计算综合排名。

第四张图是灾害影响分组条形图。用户可以在模型暴露人口、报告死亡、报告受灾人数和名义经济损失之间切换，事件使用并列水平条形比较。报告值、模型估计和缺失值使用明确标签，缺失字段保留空位并说明原因。经济损失显示原始货币年份，不能直接解释为已调整后的现实价值。

组件顶部显示事件名称、年份、海盆、数据能力和移除按钮。最多同时比较三场，以保证轨迹、曲线、标签和单位可读。悬停任一图表中的事件后，同一事件在其余三张图中高亮；点击事件将其设为 `selectedStormId` 并驱动主地图定位。

选择事件可以来自全球地图、经典案例列表或路线匹配结果。组件维护独立的 `comparisonStormIds` 集合；只有用户点击某一事件时才更新 `selectedStormId`。字段缺失时保留对应曲线断点、指标空位或影响条形空位，让用户看出数据不完整，而不是删除整行造成事件看似完整。

### 8.14 核心分析地图

`MapView.tsx` 和 `GlobeStage.tsx` 保持地图为全系统的视觉中心，并根据 `storyStage` 调整默认图层组合。全球阶段以历史轨迹、密度或聚合结果为主；单场阶段突出选中轨迹、当前位置和既有风场；历史比较阶段叠加输入路径和候选轨迹；影响阶段显示事件影响范围；区域阶段定位台湾并显示行政区；响应阶段增加设施和覆盖结果。

全球低缩放级别不应同时绘制全部高精度轨迹点，可使用简化轨迹、分箱密度或视口聚合。单条轨迹宽度可编码风速，已发生部分使用高亮实线，后续部分或完整历史轨迹使用较低透明度。选中事件必须同时依靠亮度、线宽或外描边突出，不能只改变颜色。

台湾行政区填色一次只表达一个连续指标，图例标题必须包含指标名称、单位和统计层级。设施点使用颜色与形状双重编码。地图点击结果通过共享状态传出，不在弹窗中复制右侧面板的全部内容。所有模式切换都要清理不再使用的source、layer和事件监听，避免旧图层残留形成错误叙事。

### 8.15 设施地图编辑与覆盖图层

`FacilityScenarioLayer.tsx` 在响应阶段叠加真实设施、模拟设施、服务范围和未覆盖区域。真实避难、医疗和救援设施分别使用圆形、十字和环形等固定符号；模拟设施沿用类型符号，但增加虚线外圈或“模拟”标识，确保真实数据与用户方案不会混淆。

新增设施采用明确的编辑模式：用户先选择设施类型，再点击地图放置。选中模拟设施后显示拖动手柄和小型编辑面板，可修改容量、移动或删除。拖动过程中只更新临时位置，松开后再触发评估；评估期间服务范围使用加载状态，失败时恢复到上一有效位置。

服务范围根据实际能力显示为道路服务区或距离缓冲区，并在图例中标出计算方法。已覆盖高风险区域使用低透明度蓝绿色，未覆盖区域使用橙红描边，新方案新增覆盖部分使用短暂亮色高亮。建设前后切换保持视口、阈值、行政区底色和图例范围完全一致，使视觉差异确实来自设施方案变化。

## 9. 跨组件联动约定

台风强度时间曲线、风场与地图通过 `currentTime` 联动。任何组件改变时间时只更新该字段。

风险排名、风险指标卡、缺口表和地图通过 `selectedZoneId` 联动。任何列表或图表选择行政区时只更新该字段，地图自行决定如何定位和高亮。

路线匹配列表、候选比较图和候选轨迹图通过 `selectedMatchId` 联动。选择候选时只更新该字段，各组件自行响应。

建设前后数字比较和地图覆盖结果通过 `scenarioView` 联动。比较组件负责切换该字段，地图负责显示对应图层。

页面分析尺度通过 `storyStage` 控制现有模块与地图图层组合。各图表不自行改变分析尺度。

## 10. 交付与集成方式

每个组件应在独立分支或独立提交中完成。提交内容包含组件代码、样式、必要类型、测试和验证说明。组件进入集成前必须能够在给定输入下独立渲染，并覆盖正常、加载、空数据和错误状态。

集成时只连接共享状态和公共数据类型，不在集成提交中重新修改组件内部设计。若接口不满足组件需求，应先更新接口约定，再由原负责人继续修改自己的组件。

最终验证由各组件负责人先完成专项检查，再运行项目统一验证命令：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\verify.ps1
```

组件集成后的端到端验收路径为：从全球历史轨迹、统计概览或海盆对比中选择任意事件，进入单场过程，比较历史相似案例、多个事件和事件影响；当所选事件具备台湾区域数据时，继续进入行政区风险、设施缺口和响应方案分析。路线匹配也应能够从全球历史候选进入相同的后续分析流程。
